## 四、算法

接下来是算法部分，比较复杂，而且受限于表达能力，有些地方我可能表达的不太好。

如果你同样研究过对方科目分析的算法，可以看一下；如果你对算法不感兴趣，到这里也可以不用看了，因为确实比较烧脑和痛苦。（我写的时候都感觉有点痛苦）

下面，我来分享一下具体的算法，看下如何解决自洽性问题，我把算法分为以下几步走：

- **1.漏斗分流**
- **2.穷举计算**
- **3.奥卡姆得分&记忆得分**
- **4.记忆指纹泛化**

初看，不知道说的啥？那肯定，因为算法名字都是我瞎创的，下面来逐一解释，同时我会附上对应的会计分录来展示效果。

### 1. 漏斗分流

众所周知，会计分录有非常多、非常多的类型，它们的长相千奇百怪。我所做的第一步，就是分类。

分类也是只按结构来分类，分为以下5类：

- 1借1贷型(1对1)
- 1借m贷、1贷n借型(1对多)
- n借m贷型(多对多)
- 全借全贷型(借方正数+借方负数)
- 特殊会计分录

这5类，对应分别不同的处理方式：

#### 1. 1借1贷

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 |
| :--- | :--- | :--- |:--- |
| 原材料 | 100.00 |  |应付账款 |
| 应付账款 |  | 100.00 |原材料|

对方科目直接取对方，这不必多说，最简单。

#### 2. 1借m贷、1贷n借(1对多)

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 |
| :--- | :--- | :--- |:--- |
| 原材料 | 100.00 |  |应付账款 |
| 应交税费 | 13.00 |  |应付账款 |
| 应付账款 |  | 113.00 |？|

这种稍微有点难度，但也不难，首先：
- 有多个科目的一方，它们的对方科目直接等于对方为1个的科目
- 只有1个科目的一方，它的数量直接裂变为对方一方的数量

我解释的可能不太好，看下面的例子就明白了：

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 |
| :--- | :--- | :--- |:--- |
| 原材料 | 100.00 |  |应付账款 |
| 应交税费 | 13.00 |  |应付账款 |
| 应付账款 |  | 100.00 |原材料|
| 应付账款 |  | 13.00 |应交税费|

2借1贷，属于1贷n借类型，首先，借方的对方科目直接等于应付账款，应付账款裂变为2个，分别对应借方的两个科目。

#### 3. n借m贷

如果说1v多类型的是小试牛刀，那n借m贷才是真正的地狱级难题。它的难度占对方科目分析的99%以上。

对于n借m贷，我说用到的就是**穷举算法**，这个算法在下面细说。

#### 4. 全借全贷

全借全贷是个啥呢？它其实也很常见，就长这样：

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 | 
| :--- | :--- | :--- |:--- |
| 原材料 | 100.00 |  |
| 应交税费 | 13.00 |  |
| 应付账款 |-113.00  |  

就来个借(贷)方全是正数，再来个借(贷)方全是负数。

这种怎么处理？如果检测到是属于全借全贷型的，将**负数部分移位至另一方**。

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 | 
| :--- | :--- | :--- |:--- |
| 原材料 | 100.00 |  |
| 应交税费 | 13.00 |  |
| 应付账款 |  |  113.00|

对于上面那个，就变成这样，把负数的部分移动到另一方去，然后它就变成了标准的借贷结构，再按上述三类方法处理。

#### 5. 特殊会计分录

下面我把这些类型的会计分录，定义为特殊类型：

- 结转期末损益类
- 结转期末汇兑损益类

结转损益类：

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 | 
| :--- | :--- | :--- |:--- |
| 营业收入 | 100.00 |  |
| 营业成本 |  | 13.00 |
| 管理费用 |  |  113.00|
| ... | ... |  ...|
| 本年利润 |  |  ...|

这种分录往往是多个借方、多个贷方，但解析这种分录的对方科目毫无意义，所以，这类分类我直接强制定义，对方科目全部为`本年利润`。

结转期末汇兑损益类：

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 | 
| :--- | :--- | :--- |:--- |
| 应收账款 | 100.00 |  |
| 应付账款 |  | 13.00 |
| 合同资产 |  |  113.00|
| ... | ... |  ...|
| 财务费用 |  |  ...|

这种分录也是多个借方、多个贷方，而且有都全部指向一个`财务费用`，所以，如果检测到属于结转期末汇兑损益型，则对方科目强制定义为`财务费用`，财务费用的对方科目定义为`汇兑损益调整对象`。

**怎么检测的？** 看个小算法：

- **核心attention**：如果会计分录里，只有财务费用一个损益科目，则很可能是结转汇兑损益。如果有多个损益科目，有财务费用、管理费用之类的，则财务费用必然不是主角，所以不会是结转汇兑损益。

- **三个往来**：一个分录里，至少要包含3个往科目+1个财务费用，这些关键词`["应收", "应付", "预收", "预付", "借款", "现金"]`含个3上，才有可能判定为结转汇兑损益，

- **白名单剔除**：`["应付职工", "应交税", "应付股","应付利"]`这些关键词，跟财务费用在一起的，不判定为结转汇兑损益。

### 2. 穷举计算

上面说了，只有n借m贷型才会进入穷举计算，其他的类型，在传入序时账时，代码就已经刷刷解决掉了。

穷举计算的意思：对于一个n借m贷型，尝试各种组合，列示出所有可能的组合。比如几个点，可能通过各种连线组合，发展为**枝繁叶茂的树**。

但是，我们不是100%列出所有可能的组合，而是要添加约束条件，也叫做**剪枝**。

我们直接来个分录：

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 | 
| :--- | :--- | :--- |:--- |
| 应收账款 | 15,000.00 |  |
| 合同负债 | 12,000.00 |  |
| 营业收入 |  |  11,000.00|
| 银行存款 |  |  14,000.00|
| 应交税费 |  |  2,000.00|

这是一个2借3贷的会计分录，要怎么计算？我们来肉眼看一下，应收账款`15,000.00`可能由哪些贷方构成？
简单列示一下：
```test
组合1：✅️
[营业收入:11,000.00,
银行存款:4,000.00]
组合2：✅️
[营业收入:11,000.00,
银行存款:2,000.00,
应交税费:2,000.00]
组合3：❌️
[银行存款:15,000.00]
组合4：❌️
[营业收入:1000.00,
银行存款:13,000.00,
应交税费:1,000.00]
....
```
我们尝试算出这个分录可以有的所有组合。为什么有些组合我打上了"✅️"，有些打上了"❌️",下面我们来看约束条件：

### 2.1 穷举计算的约束条件

上个例子里，我们对应收账款借方`15,000.00`求解可能有的贷方组合。

穷举，肯定不是盲目地算，而是有约束条件。我设置了3项约束条件：

####  1.上限不可越
**科目本身上限**：求解的组合合计金额，不能超过应收账款借方`15,000.00`

**对方科目上限**：参与求解的科目，不能超过它自身的上限；我们注意看：
`组合3`:`[银行存款:15,000.00]`

贷方银行存款最大值也只有14,000，所以这个组合是错误的，我给它打上了个❌️。

#### 2.边界计算：穷举组合中至多出现1项非边界值

什么是边界计算？科目的全部值，就是边界值。

我们看一下应收账款借方`15,000.00`的组合1、组合2：

哪个是边界值？哪个是非边界值：
```test
组合1：✅️
[营业收入:11,000.00, <-- = 贷方营业收入金额11,000.00，是边界值
银行存款:4,000.00] <-- ≠ 贷方银行存款金额14,000.00，是非边界值
组合2：✅️
[营业收入:11,000.00,<-- = 贷方营业收入金额11,000.00，是边界值
银行存款:2,000.00,<-- ≠贷方银行存款金额14,000.00，是非边界值
应交税费:2,000.00]<-- = 贷方应交税费金额2,000.00，是边界值
....
```
组合1、组合2中，都只有1项非边界值，所以是正确的，组合4呢？

```test
组合4：❌️
[营业收入:1000.00, <-- ≠ 贷方营业收入金额11,000.00，是非边界值
银行存款:13,000.00, <-- ≠贷方银行存款金额14,000.00，是非边界值
应交税费:1,000.00]<-- ≠ 贷方应交税费金额2,000.00，是非边界值
```
可以看到，组合4有3项非边界值，不满足只有1项的规则，所以是错误的。

**非边界值的实质是什么？为什么要设置至多只能由1项非边界值？**

如果不设置至多只能有1项非边界值，那么，应收账款借方`15,000.00`可以有无数个组合：

```test
组合4：
[营业收入:1000.00,
银行存款:13,000.00,
应交税费:1,000.00]
组合5：
[营业收入:999.00,
银行存款:12,999.00,
应交税费:1,002.00]
组合6：
[营业收入:998.00,
银行存款:12,998.00,
应交税费:1,004.00]
...
```
看到了吗，如果不设置非边界值至多至有1项，对方科目的金额可以随便取，反正只要能凑够`15,000.00`就行。

还有，非边界值至只能出现1项的实质就是：**组合只能轧差轧1次。**

另外，我们还可以从**自由未知量**的概念来解释，假设一个科目金额`i`由对方科目中n个科目构成，则有：$$i = x_{1}+x_{2}+x_{3}+...+x_{n}$$

在这一项公式中，自由未知量只能允许有`1个`，如果有`2个`以上会发生什么？

例如，公式里有2个自由未知量$$x_{1}、x_{2}$$，
$$x_{1} =1，x_{2} = 9 或x_{1}=2 ，x_{2}=8$$，都满足$$x_{1}+x_{2}$$不变，代入原公式里都满足：
$$i = x_{1}+x_{2}+x_{3}+...+x_{n}$$
所以$$x_{1}和x_{2}$$可以随便组合，`i`会变成无限个。

那无限多个，就穷举失败了。

因此，自由未知量只能有1个，即组合只能轧差1次。

#### 3.边界值动态更新

边界值是可以动态更新的，我们以组合2为例：

组合2取走了：`[营业收入:11,000.00,银行存款:2,000.00,应交税费:2,000.00]`

| 一级科目 | 贷方金额 | 已取走金额 | 剩余金额 | 
| :--- | :--- | :--- |:--- |
| 营业收入 | 11,000.00 |  11,000.00| 0.00
| 银行存款 | 14,000.00 |  2,000.00| 12,000.00
| 应交税费 |  2,000.00 |  2,000.00| 0.00

此时，银行存款的边界值会更新，由原来的金额`14,000.00`变为`12,000.00`，继续以这个新的边界值参与穷举计算。

### 2.2 穷举计算的算法规则

#### 1.谁遍历谁？

| 一级科目 | 借方金额 | 贷方金额 | 对方科目 | 
| :--- | :--- | :--- |:--- |
| 应收账款 | 15,000.00 |  |
| 合同负债 | 12,000.00 |  |
| 营业收入 |  |  11,000.00|
| 银行存款 |  |  14,000.00|
| 应交税费 |  |  2,000.00|

对于一个标准的借贷结构会计分录，我们只需要求出其中一方的组合方式，那另一方就不用求了。

问题就是，依据借方，去求贷方的组合；还是依据贷方，去求借方的组合？

我的规则：**“对于一个n借m贷的会计分录，当n＜m时，用m遍历n；当n≥m时，用n遍历m”**

对于这个会计分录来说，2借3贷，谁的个数多，依据谁来求组合，那就是求贷方可能的组合。

比如说，我们从贷方出发，求营业收入可能有的组合，遍历对象无非就2个，要么应收账款、要么合同负债。

而如果我们从借方出发，求应收账款可能有的组合，那就是3个，营业收入、银行存款、应交税费都有可能。

如果借贷方数量差异大起来，那组合数的差异是非常明显的。

比如说，一个2借7贷的会计分录，从借方走，有7个遍历对象，组合数会爆炸；但是从贷方走，仍然是只有2个遍历对象，组合数会相对较小。

大概的道理就是：$$2^{7}$$大还是$$7^{2}$$大？

#### 2. 谁先开始遍历--最小资源占优

对于上个会计分录，我们决定了，以贷方遍历借方。那看贷方，有3个会计科目，营业收入、银行存款、应交税费，那么，谁先开始？

**最小资源占优原则**：最小的开始，依次排序。

也就是说，应交税费最小，先确定应交税费的组合，再去确定营业收入的组合、银行存款的组合。

这一条原则对于穷举计算非常关键，我们再看一个例子：

| 一级科目 | 借方金额 | 贷方金额 |
| :--- | :--- | :--- |
| 管理费用 | -292.24 | - |
| 销售费用 | -92.58 | - |

### 3. 奥卡姆得分&记忆得分

#### 1. 奥卡姆得分

在穷举计算完成后，我们计算出了所有可能的组合，还有个重要问题，又来了：

**我们怎么选出正确的组合？**

比如，这个会计分录，程序给我列了一堆组合，那我该选哪个？

![](https://fastly.jsdelivr.net/gh/bucketio/img6@main/2025/12/28/1766899845886-b615c5cb-c1c2-45ad-973f-855d1fed1674.png)

这时，就要引入**奥卡姆剃刀原理**了，就一句话：**如无必要，勿增实体。**

翻译就是：怎么简单怎么来。

也就是，我们就选最简单的一个组合，最简单的往往是对的，因为会计记账时，往往就是简单记账的，他们没理由会乱记、交叉着瞎记。

那么，怎么定义最简单？我们设置一个奥卡姆得分，每个组合初始得分为100分。

在穷举计算中，会产生科目与科目之间的多个连接，我们的目的，就是找出连接数最少的一个组合。

- **奥卡姆得分的计算规则**：
- 1.每连接1次，扣1分
- 2.还记得遍历规则吗？**对于一个n借m贷的会计分录，当n＜m时，用m遍历n；当n≥m时，用n遍历m**；如果是遍历方，被拆分，扣5分；如果是被遍历方，被拆分，扣1分；

比如说，这个会计分录：

| 一级科目 | 借方金额 | 贷方金额 |对方科目|
| :--- | :--- | :--- |:---|
| 管理费用 | 100.00 | - |
| 销售费用 | 100.00 | - |
| 研发费用 | 100.00 | - |
| 应付职工薪酬 | - | 50.00 |
| 银行存款 | - | 250.00 |

这是3借2贷分录，哪边多，哪边开始遍历，用借方遍历贷方：

如果是贷方银行存款被拆分：

| 一级科目 | 贷方金额 | 被拆分情况 |对方科目|
| :--- | :--- | :--- |:---|
| 银行存款 | 250.00 | 50.00 |管理费用
| 银行存款 |  | 100.00 |研发费用
| 银行存款 |  | 100.00 |销售费用

贷方，是被遍历方，被遍历方被拆分1次，只扣1分。

如果是借方销售费用费用被拆分：

| 一级科目 | 贷方金额 | 被拆分情况 |对方科目|
| :--- | :--- | :--- |:---|
| 销售费用 | 100.00 | 50.00 |应付职工薪酬
| 销售费用 |  | 50.00 |银行存款

借方，是遍历方，遍历方被拆分1次，扣5分。

特殊的“硬骨头”：比如，应交税费这个科目就是个“硬骨头”，这个科目通常不会被拆分，所以，我给这些硬骨头设置了更高的扣分权重，如果这个科目被拆分，扣分情况是其他科目的2倍。

所以，我们这里就可以得出奥卡姆得分的计算公式：

$$
奥卡姆得分 = 100 - 连接数(行数)*1 - 遍历方拆分次数*5 - 被遍历方拆分次数*1 - 硬骨头拆分惩罚
$$
代码实现情况可以看`occams_razor.py`里的`score_solution`函数。

#### 2. 记忆得分

什么是记忆得分？其实跟我下面要说的**记忆指纹匹配**有关。

奥卡姆得分可能能解决大部分分录的组合，但可能总会有错误的，比如有些分录还真不是越简单越正确，可能它就是连接很复杂。

这怎么办呢？只能加入**人工复核。**

在运行对方科目分析之后，程序会自动选择它认为最可能的一组组合，它选择的依据就是奥卡姆得分最高者。

它会自动给得分最高者的方案打上"x"，表示选中该方案：

![](https://fastly.jsdelivr.net/gh/bucketio/img13@main/2025/12/28/1766906889838-99882fd8-941e-4817-ad58-2710fb9fb3e0.png)

但是，未必是正确的，比如上图中，正确的方案应该是`46-2`，而不是程序选择的`46-1`。

那怎么办？直接手动更改，取消`46-1`的"x"，手动打在`46-2`上。

![](https://fastly.jsdelivr.net/gh/bucketio/img17@main/2025/12/28/1766907039956-8cb3663c-db91-4b68-b764-3358861e0c47.png)

这样就行。在你纠错之后，程序就会知道，噢，原来46-2才是对的，后面它遇到类似的，都会选择46-2的结构。

那么，程序是如何知道你教它的东西的呢？就是**记忆得分（memory）**。

直接列出记忆得分的公式：

$$
M_{new} = M_{old} * (1 - a) + reward * a
$$

参数说明：
- $$M_{old}$$：上一轮的记忆值，初始记忆值默认为`0.5`
- $$reward$$：选中奖励，选中为1，没选中为0
- $$a$$：衰减率或学习率，目前我设置的默认值是`0.6`

演示一下：
- **第一次选择**：
被选中的组合A的M值：
$$ = 0.5 * (1-0.6) + 1 * 0.6 = 0.8$$
没被选中的组合B的M值：
$$ = 0.5 * (1-0.6) + 0 * 0.6 = 0.2$$

- **第二次，需要更换组合，由组合A更换为组合B**：
新组合的M值：
$$= 0.2 * (1-0.6) + 1 * 0.6 = 0.68$$
旧组合的M值：
$$= 0.8 * (1-0.6) + 0 * 0.6 = 0.32$$

可以看到，记忆得分就是这么地“喜新厌旧”，每次如果更换了组合，之前没被选中的组合会瞬间反超，变成新的“香饽饽”。

#### 3. 合计得分
合计得分就是把奥卡姆得分和记忆得分加起来，公式是这个：

$$
TotalScore=奥卡姆得分 * (1+β×M)
$$
β我设置的默认值是`0.5`

最终合计得分最高者，会被程序默认选中。


### 4. 记忆指纹泛化

#### 1. 什么是指纹？

在我们把方案导入回程序以后，程序会生成一个`contra_memory_ema.json`文件，存储在`user_data`目录下，这个其实就是程序的“记忆”。

比如说这一段json内容：
```json
  "固定资产、固定资产清理、累计折旧、营业外支出": {
    "固定资产清理->固定资产清理|累计折旧->固定资产|营业外支出->固定资产": 0.0253,
    "固定资产清理->固定资产|固定资产清理->固定资产清理|累计折旧->固定资产清理|营业外支出->固定资产清理": 0.0003,
    "固定资产清理->固定资产|累计折旧->固定资产|营业外支出->固定资产清理": 0.9747,
    "固定资产清理->固定资产|累计折旧->固定资产清理|营业外支出->固定资产|营业外支出->固定资产清理": 0.0003,
    "固定资产清理->固定资产|累计折旧->固定资产|累计折旧->固定资产清理|营业外支出->固定资产": 0.0003
```

其中：
`"固定资产清理->固定资产|累计折旧->固定资产|营业外支出->固定资产清理": 0.9747`

这里的0.9747，就是这个组合的记忆得分，它最高分。

就是说，固定资产清理要跟固定资产连接、累计折旧要跟固定资产连接....

比如说这个多借多贷的会计分录：

| 一级科目 |   借方金额 | 贷方金额 |
| :--- |  :--- | :--- |
| 固定资产清理 |   1,628.31 |  |

| 营业外支出 |   1,628.31 |  |

下次，程序遇到了这样的多借多贷会计分录，它就知道，要按它记忆里得分最高的这个匹配模式来处理。

分录长什么样，就是**指纹**。

#### 2. 什么是泛化？

就是有相同指纹的会计分录，全部都适用于这套分配模式。

记忆泛化的好处就是，你教一遍之后，它就马上记住。对于这一套序时账，只要会计记账方式没有发生很大变动，那每年都可以适用这一套记忆。

### 1. 总结
总结一下大概的流程，传入序时账后：
1. **分类处理**：区分5类情况进行处理
2. **穷举计算**：计算出多借多贷可能有的所有组合
3. **筛选组合**：根据奥卡姆得分筛选组合
4. **人工复核**：人工复核程序筛选的组合
5. **机器记忆**：程序将每个组合的解法记录下来
6. **指纹泛化**：程序将组合解法应用到整个序时账
