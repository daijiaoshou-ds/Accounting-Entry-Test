# 会计分录异常检测系统 - 技术设计文档

## 一、理论设计

### 1.1 核心概念

#### 1.1.1 凭证唯一值
```
格式：{年份}年{月份}月-{原凭证号}
示例：2025年9月-记-0497

数学表达：
VoucherID = Year(Date) & "年" & Month(Date) & "月-" & OriginalNo
```

#### 1.1.2 会计分录特征
```
定义：凭证涉及的所有一级科目，去重后按"、"连接

示例凭证：
  借：应收账款 100
  借：应交税费 20
  贷：主营业务收入 100
  贷：应交税费-销项税 20

会计分录特征：应收账款、应交税费、主营业务收入

数学表达：
Feature(V) = Sort(Unique({Subject | Subject ∈ Voucher V}))
```

#### 1.1.3 对方科目
```
定义：当前分录行对应的反向科目

规则：
- 当前行为借方时，对方科目 = 该凭证所有贷方科目
- 当前行为贷方时，对方科目 = 该凭证所有借方科目

数学表达：
Counter(i) = {Subject | Subject ∈ Voucher V(i), Direction(Subject) ≠ Direction(i)}
```

### 1.2 群聚类理论

#### 1.2.1 核心群定义
```python
CORE_GROUPS = {

    '长期资产群': ['固定资产', '在建工程', '无形资产', '累计折旧'],

```

#### 1.2.2 聚类规则（新版）

**特性群判定（优先级最高）**
```
特性群 = [薪酬活动群, 长期资产群, 税务活动群, 生产活动群, 资金活动群]

IF 任一科目 ∈ 特性群:
    PrimaryGroup = 该特性群
    InvolvedGroups += 该特性群
```

**非特性群二元锚定（采购群、销售群）**

*一借一贷型（|Subjects| = 2）*
```
A. 两个科目都踩中同个群:
   如：应付账款、原材料（都踩中采购群）→ 归入采购群

B. 两个科目分别踩中不同群:
   如：应收账款（销售群）、原材料（采购群）→ 跨销售群+采购群
   
C. 只有一个科目有匹配:
   归入该群
   
D. 都不匹配:
   归入其他群
```

*n借m贷型（|Subjects| ≥ 2）*
```
至少需要踩中2个核心词才归该群:

如：应付账款、原材料、应交税费
- 采购群：应付账款+原材料 = 2个词 ✓ 归入采购群
- 税务群：应交税费 = 1个词 ✗（但税务群是特性群，优先判定）

如：应付账款、原材料、应收账款
- 采购群：应付账款+原材料 = 2个词 ✓
- 销售群：应收账款 = 1个词 ✗
- 结果：归入采购群

如：应付账款、原材料、应收账款、主营业务成本
- 采购群：应付账款+原材料 = 2个词 ✓
- 销售群：应收账款+主营业务成本 = 2个词 ✓
- 结果：跨采购群+销售群
```

### 1.3 群距离理论

#### 1.3.1 共现率
```
定义：两个群之间跨界凭证的比例（平滑处理）

公式：
CrossRate(A,B) = (CrossCount(A,B) + 1) / (Count(A) + Count(B) + 2)

其中：
- CrossCount(A,B) = 同时涉及群A和群B的凭证数量
- Count(A) = 归属群A的凭证数量
- +1 和 +2 为拉普拉斯平滑
```

#### 1.3.2 群距离
```
公式：
Distance(A,B) = -ln(CrossRate(A,B))

性质：
- 当跨界凭证多 → CrossRate大 → Distance小（群相近）
- 当跨界凭证少 → CrossRate小 → Distance大（群相远）
```

### 1.4 异常检测理论

#### 1.4.1 群距离计算（新版，支持k群跨界）
```
共现率计算（区分k值）：
对于涉及k个群的跨界凭证组合 (G1, G2, ..., Gk)：
x_k = (count_k + 1) / (count_G1 + count_G2 + ... + count_Gk + 2)

总共现率：x = Σ x_k  （对所有包含该群对的k群组合求和）

距离计算（倒数距离）：
D(Gi, Gj) = min(1/x - 1, 100)

示例：
总凭证300笔（工资100、销售100、采购100）
- 采购&工资跨界30笔，采购&销售跨界15笔，采购&销售&工资跨界5笔

x(采购,工资) = (30+1)/(100+100+2) + (5+1)/(100+100+100+2) = 0.154 + 0.020 = 0.174
D(采购,工资) = min(1/0.174 - 1, 100) = min(4.75, 100) = 4.75
```

#### 1.4.2 跨模块异常得分
```
对于跨群的凭证V（涉及群G1, G2, ..., Gn）：

Score_cross(V) = Σ Distance(Gi, Gj) + Penalty_other

惩罚机制：
- 跨了其他群（其他群+某群）：+10分
- 完全归入其他群：+20分
```

#### 1.4.3 模块内异常得分（倒数距离版）
```
对于在群G内的凭证V，其特征为F：

Freq(F) = Count(F in G) / Count(G)
Score_inner(V) = min(1/Freq(F) - 1, 100) + Penalty_other

惩罚机制同上

特殊情况：
- 如果 Freq(F) = 0（从未出现）: Score_inner = 100
- 如果 Freq(F) 极低: Score_inner ≈ 100（倒数会很大）
- 如果 Freq(F) 正常: Score_inner ≈ 0
```

#### 1.4.4 综合风险得分
```
TotalScore(V) = Score_cross(V) + Score_inner(V)

风险等级（新标准）：
- TotalScore ≥ 80: 高风险 🔴
- 30 ≤ TotalScore < 80: 中风险 🟠
- 10 ≤ TotalScore < 30: 中低风险 🟡
- TotalScore < 10: 正常 🟢
```

### 1.5 科目连接图理论

#### 1.5.1 图的构建
```
节点：群G内所有出现的科目（去重）

边：贷方科目 → 对方科目（借方科目）

边权重：W(A→B) = |{Voucher | Voucher包含(A贷方, B借方)}|
         即：有多少个凭证包含这条资金流动

重要：权重按凭证计数，不是按行计数！
```

#### 1.5.2 图的特性
```
1. 资金流向：从贷方流向借方（会计恒等式）
2. 不走回头路：应收账款→银行存款→应付账款（单向）
3. 汇聚性：多个科目可能汇聚到同一科目（如多借一贷）
```

```python
        to_subjects = row['counter_subject'].split('、')
        
        for to_subject in to_subjects:
            edge_vouchers[(from_subject, to_subject)].add(voucher_id)
    
    # 5. 计算权重 = 凭证数量
    edges = {edge: len(vouchers) for edge, vouchers in edge_vouchers.items()}
    
    return nodes, edges
```

## 三、数学计算详解

### 3.1 群距离计算示例

假设数据：
- 采购活动群：100个凭证
- 销售活动群：80个凭证
- 跨界凭证（同时涉及采购和销售）：5个

计算：
```
CrossRate(采购, 销售) = (5 + 1) / (100 + 80 + 2) = 6/182 ≈ 0.033

Distance(采购, 销售) = -ln(0.033) ≈ 3.41

解释：距离3.41表示采购和销售是较远的业务群
（正常的跨界凭证应该很少）
```

### 3.2 异常得分计算示例

#### 示例1：跨模块异常
凭证特征：原材料、主营业务收入、应收账款、应付账款
涉及群：采购活动群 + 销售活动群

假设：
- Distance(采购, 销售) = 3.41

计算：
```
Score_cross = 3.41
Score_inner = 0（假设该特征在采购群出现过）

TotalScore = 3.41 + 0 = 3.41 → 低风险 🟢
```

#### 示例2：模块内低频异常
凭证特征：应收账款、营业外支出
归属群：销售活动群（因为有应收账款）

假设：
- 销售活动群共1000个凭证
- "应收账款、营业外支出"特征只出现1次

计算：
```
Freq = 1/1000 = 0.001
Score_inner = -ln(0.001) = 6.91
Score_cross = 0（不跨群）

TotalScore = 6.91 → 中风险 🟡
```

#### 示例3：严重异常
凭证特征：固定资产、应付职工薪酬
涉及群：长期资产群 + 薪酬活动群

假设：
- Distance(长期资产, 薪酬) = 8.5（非常远）
- 该特征从未出现过

计算：
```
Score_cross = 8.5
Score_inner = 10（从未出现）

TotalScore = 18.5 → 高风险 🔴
```

### 3.3 连接图权重计算示例

数据：
```
凭证1：贷：主营业务收入 -> 借：应收账款
凭证2：贷：主营业务收入 -> 借：应收账款

```

```
- 应交税费 → 银行存款: 1个凭证

注意：即使凭证1有100行，权重也是1不是100！
```

## 四、关键设计决策

### 4.1 为什么过滤本年利润？
```
原因：
1. 结转本年利润是正常的期末操作
2. 几乎所有企业都会做，不具有异常检测价值
3. 涉及"本年利润"的凭证往往行数很多，干扰统计

过滤规则：
会计分录特征包含"本年利润"或"利润分配"的凭证，直接过滤
```

### 4.2 为什么使用凭证计数？
```
原因：
1. 会计分录测试的本质是检查"凭证"的合理性
2. 一个凭证可能有1行也可能有1000行，但都是一个业务事项
3. 使用行数会导致大凭证（如工资计提）权重过高

实现：
所有统计都先按voucher_unique_id去重，再计数
```

### 4.3 为什么对方科目可以用户上传？
```
原因：
1. 很多财务软件导出的数据已经包含对方科目
2. 用户上传的对方科目更准确（考虑了辅助核算等因素）
3. 如果用户没有上传，系统可以自动计算作为备选

实现：
配置页面增加"对方科目列"选项，默认"自动计算"
```

### 4.4 为什么使用Sankey图？
```
原因：
1. Sankey图天然适合展示资金流向
2. 线宽可以表示流量大小（凭证数量）
3. 可以清晰展示多对一、一对多的汇聚关系

替代方案：
- 有向图：节点太多时混乱
- 矩阵图：无法展示流向
```

## 五、机器学习补充分类（MLClassifier）

### 5.1 设计目的

当规则无法对凭证进行分类时（如科目不在任何核心群中），使用机器学习进行补充分类。

**适用场景**：
- 凭证特征中的科目都不在预定义的核心群中
- 每个核心群最多只匹配1个科目（不满足n借m贷型的≥2要求）
- 规则分类结果为"其他群"

### 5.2 算法原理

#### 5.2.1 TF-IDF向量化
```python
# 1. 将会计分录特征转换为文本
feature = "应收账款、主营业务收入"
text = "应收账款 主营业务 收入"

# 2. TF-IDF向量化
vectorizer = TfidfVectorizer(
    max_features=100,    # 最多100个特征词
    min_df=2,            # 至少出现在2个文档中
    ngram_range=(1, 2)   # 使用1-gram和2-gram
)

# 输出：稀疏向量 (n_samples, n_features)
X = vectorizer.fit_transform(texts)
```

#### 5.2.2 群代表向量构建
```python
# 对每个已分类的群，计算其平均向量（质心）
for group in groups:
    group_vectors = X[group_mask]  # 该群所有凭证的向量
    centroid = np.mean(group_vectors, axis=0)  # 质心
    group_centroids[group] = centroid
```

#### 5.2.3 相似度计算与预测
```python
# 对待分类凭证：
# 1. 向量化
test_vector = vectorizer.transform([test_feature])

# 2. 计算与各群质心的余弦相似度
for group, centroid in group_centroids.items():
    similarity = cosine_similarity(test_vector, centroid)
    
# 3. 选择相似度最高的群（需超过阈值0.3）
best_group = argmax(similarity)
if similarity[best_group] >= 0.3:
    prediction = best_group
else:
```
